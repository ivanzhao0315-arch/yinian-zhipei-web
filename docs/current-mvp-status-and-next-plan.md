# 颐年智陪当前 MVP 状态与下一阶段计划

## 1. 当前结论

颐年智陪已经完成一个可运行的本地 MVP 原型。

当前版本不是生产系统，数据使用本地 SQLite 做开发期持久化，但已经跑通核心业务闭环：

```text
家属提交预约
-> 运营后台确认订单
-> 运营后台指派陪诊员
-> 陪诊员端更新服务进度
-> 陪诊员端记录异常
-> 运营后台处理异常
-> 陪诊员完成服务
-> 家属端查看服务总结
```

## 3. 当前运行方式

启动本地 API：

```bash
npm run dev:api
```

启动运营后台：

```bash
npm run dev:admin
```

启动小程序构建监听：

```bash
npm run dev:mini-program
```

本地地址：

```text
运营后台：http://127.0.0.1:5174
本地 API：http://127.0.0.1:5175
```

微信开发者工具打开：

```text
apps/mini-program
```

开发阶段需要在微信开发者工具里关闭“校验合法域名”，否则小程序无法请求本地 `127.0.0.1:5175`。

## 5. 当前限制

### 5.1 非真实登录

当前没有接入：

- 微信登录。
- openid。
- 手机号授权。
- 运营后台账号密码。
- 陪诊员身份绑定。

当前已通过请求头模拟家属、运营和陪诊员身份，并做了基础权限隔离：

- 家属只能看自己的订单。
- 运营后台接口需要 admin 演示身份。
- 陪诊员只能看和更新分配给自己的任务。

这只是开发期权限边界，不等同于真实登录。

已预留真实登录所需的数据结构和开发接口：

- `users`：家属微信 `open_id`、昵称、手机号。
- `admin_users`：运营后台账号。
- `escort_openid_bindings`：陪诊员微信 `open_id` 与自营陪诊员 ID 的绑定关系。
- `/api/auth/demo/wechat-login`：开发期模拟微信登录。
- `/api/admin/auth/demo-login`：开发期模拟后台登录。
- `/api/escort/auth/demo-bind`：开发期模拟陪诊员微信绑定。

### 5.2 非生产级数据库

当前没有生产级数据库。

影响：

- 本地 SQLite 不适合多人协作和线上高并发写入。
- 真实用户、订单、进度和运营复盘后续建议迁移到 PostgreSQL。
- 后续需要增加数据备份、迁移和权限控制。

### 5.3 非真实支付

当前已经有：

- 微信支付 API v3 预支付接口骨架。
- 支付记录表。
- 小程序支付参数生成。
- 微信支付通知解密与支付成功落库。

当前还没有：

- 退款。
- 微信支付真实商户联调验收。
- 公网 HTTPS 回调地址。
- 财务对账。

首版可以先用 `WECHAT_PAY_MODE=mock` 本地联调。切到真实支付时需要配置商户号、小程序 AppID、商户 API 证书序列号、商户私钥文件路径、APIv3 密钥和公网 `notify_url`。

### 5.4 非真实通知

当前没有：

- 微信订阅消息。
- 短信。
- 电话外呼系统。
- 后台实时提醒。

状态变化需要用户手动刷新。

### 5.5 文件与隐私能力未接入

当前没有：

- 图片上传。
- 病历/检查单权限控制。
- 隐私授权记录。
- 敏感信息脱敏。

真实上线前必须补充。

## 6. 下一阶段建议目标

下一阶段目标不是继续加页面，而是把 SQLite 本地 API 迁移成最小生产后端。

建议目标：

```text
可支撑 10-30 单真实试运营
```

核心要求：

- 数据能保存。
- 订单能追踪。
- 运营能复盘。
- 用户隐私有基本保护。
- 关键操作有日志。

## 7. 推荐后端路线

推荐：

```text
Fastify + PostgreSQL
```

理由：

- 当前本地 API 已使用 Fastify，迁移成本低。
- PostgreSQL 适合订单、进度、异常和服务总结长期留存。
- 后续可以做运营数据分析。

已完成过渡方案：

```text
Fastify + SQLite
```

适合：

- 只做极小范围试单。
- 暂时不部署复杂数据库。

不建议首选：

```text
只用前端 localStorage
```

原因：

- 无法多人协同。
- 无法运营复盘。
- 数据容易丢。

## 8. 下一阶段开发拆分

### M1：真实数据模型

目标：

- 把当前 SQLite 数据模型迁移为 PostgreSQL 表。
- 保留当前 API 路径不变。

建议表：

- `users`
- `elder_profiles`
- `escorts`
- `orders`
- `order_progress`
- `order_exceptions`
- `service_summaries`
- `order_logs`
- `admin_users`

### M2：登录与权限

目标：

- 家属端接微信登录。
- 陪诊员绑定微信 openid。
- 运营后台账号密码登录。

权限规则：

- 家属只能看自己的订单。
- 陪诊员只能看分配给自己的任务。
- 运营可以看全部订单。

### M3：真实订单运营

目标：

- 运营后台可真实管理订单。
- 收款状态先人工记录。
- 异常与总结可持续留存。

新增字段：

- 收款状态。
- 取消原因。
- 退款备注。
- 服务总结审核状态。
- 运营复盘备注。

### M4：通知与消息

目标：

- 关键状态变化通知家属。
- 新订单提醒运营。
- 新任务提醒陪诊员。

首版建议：

- 微信订阅消息。
- 后台声音/红点提醒。
- 暂不接复杂短信平台。

### M5：隐私与文件

目标：

- 支持图片上传。
- 病历、检查单、缴费凭证权限控制。
- 敏感信息脱敏。

必须注意：

- 文件不能公开访问。
- 后台查看敏感资料需要权限。
- 操作日志要记录谁看过和谁上传。

## 9. 最近可执行的下一步

建议从这里开始：

1. 在 `apps/api` 接入 PostgreSQL 或 SQLite。
2. 把当前本地数据模型迁移为数据库表。
3. 保持当前 API 路径不变。
4. 先让后台订单池读取真实数据库。
5. 再让小程序提交预约写入数据库。

完成后，MVP 就从“可演示原型”进入“可小范围试运营系统”。
